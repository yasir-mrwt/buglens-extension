TestPilot Agent, Prompt, and Model Flow Handoff
================================================

Purpose
-------
This file explains how TestPilot handles Agent prompts, AI chat prompts, AI model calls,
permission-gated browser actions, generated test cases, generated bug reports, and report
reuse. It is intended as a technical handoff for updating the Agent architecture.


High-Level Architecture
-----------------------
TestPilot has three separate intelligence paths:

1. Agent browser workflow
   - Runs inside the inspected page through src/content/content-script.ts.
   - Uses deterministic local planning and DOM inspection.
   - Does not currently let the LLM execute arbitrary browser actions.
   - Fills fields, highlights controls, clicks safe controls, observes page state, and reports evidence.
   - Requires tester approval before submit/continue/save/send/login/signup/order/payment-style actions.

2. AI chat
   - Runs from src/devtools/panel/panelController.ts.
   - Sends sanitized QA session context to the configured AI provider.
   - For local development, the default provider is the local backend at http://localhost:8787.
   - Uses streaming local backend chat when possible.
   - Answers QA/session questions only from captured context.

3. AI artifacts
   - Analyze Session, Generate Test Cases, and Generate Bug Reports.
   - Built from sanitized session findings plus latest Agent result.
   - Can use the local backend or frontend provider adapters.
   - Falls back to deterministic local output when local backend/model fails.

Agent Browser Workflow
----------------------
Entry point:
  src/devtools/panel/panelController.ts -> runAgentCommand(command, options)

Message sent to inspected page:
  type: TESTPILOT_RUN_AGENT
  command: user text from chat
  options:
    mode
    sessionId
    pageUrl
    contextMode
    history
    approval

Receiver:
  src/content/content-script.ts -> chrome.runtime.onMessage listener
  message.type === TESTPILOT_RUN_AGENT
  calls runAgentWorkflow(command, options)

Core Agent functions in src/content/content-script.ts:
  runAgentWorkflow(command, options)
    - Sanitizes tester command.
    - Extracts current page context.
    - Detects data strategy.
    - Classifies task type.
    - Builds requestKey.
    - Builds deterministic fallback plan.
    - Normalizes any structured plan if provided.
    - Emits live Agent events.
    - Runs safe steps.
    - Pauses before approval-required steps.
    - Resumes from pending step after approval.
    - Observes page result and returns final Agent result.

  extractAgentPageContext()
    - Finds visible interactive elements.
    - Describes forms, fields, submit buttons, headings, visible alerts, and page type.
    - Builds a context hash used by requestKey.

  buildAgentPlan(command, taskType, context, options)
    - Deterministic local planner.
    - Creates steps such as clear, type, select, check, click, observe, highlight, wait, scroll.
    - Does not currently ask the LLM to produce executable browser actions.

  executeAgentStep(step, index, context, requestKey)
    - Executes one safe action.
    - Field typing uses setElementValueAnimated().
    - Emits field-updated live progress events.
    - Clicks only planned targets and waits briefly for resulting evidence.

  observeAgentPage(context)
    - Collects alerts, field validation, dialogs, result signals, buttons, accessibility observations, etc.

  evaluateAgentResult(plan, actionResults, observation, safety)
    - Converts action results and page observation into passed/failed/needs-review Agent result.


Agent Prompt Handling
---------------------
The Agent command is natural language from the tester, but it is not passed directly to an LLM for action execution.

Current Agent prompt flow:
  1. Tester types command in Agent mode.
  2. src/devtools/panel/panelController.ts submitAiChat() detects Agent mode.
  3. src/devtools/panel/panelController.ts runAgentCommand() sends command to content script.
  4. src/content/content-script.ts classifies the command with rule-based logic.
  5. src/content/content-script.ts builds a deterministic plan from DOM context.
  6. Agent executes the plan with safety gates.

Important classification functions:
  src/content/content-script.ts -> detectDataStrategy(command)
    - Determines whether the user asked for empty, invalid, dummy, valid dummy, or real user provided data.

  src/content/content-script.ts -> classifyAgentTask(command, context, dataStrategy)
    - Maps command to task type:
      required_field_validation
      invalid_login_test
      invalid_signup_validation
      login_validation
      valid_dummy_data_flow
      form_validation
      search_test
      filter_test
      modal_test
      pagination_test
      accessibility_check
      page_summary
      test_case_generation
      bug_report_generation
      table_test
      general_page_validation

Important safety point:
  The Agent does not currently rely on model-generated action plans. This reduces prompt-injection risk from page content.
  If a future upgrade adds LLM planning, keep the content-script validator as the final authority before any action runs.


Agent Permission Flow
---------------------
The Agent is designed to fill safe inputs first, then ask the tester before submitting/continuing.

Approval-required actions include:
  - Plain form Submit
  - Continue
  - Save
  - Send
  - Login / Sign in
  - Signup / Register / Create account
  - Checkout / Place order / Pay / Purchase
  - Delete / Remove / Destroy
  - Logout / Sign out
  - Reset password
  - File upload
  - External or unsafe navigation

Core safety functions:
  src/content/content-script.ts -> validateAgentPlan(plan, context, approval, requestKey)
    - Checks unavailable targets and disabled targets.
    - Finds confirmation-required steps.
    - Confirms whether approval matches the request and pending step.

  src/content/content-script.ts -> getAgentStepConfirmation(step, context)
    - Decides if a step requires approval.
    - Returns reason, category, rememberable flag, and target label.

  src/content/content-script.ts -> isAgentFormSubmitAction(step, target, context)
    - Detects form submit/continue/save/send/login/signup-style buttons.

  src/content/content-script.ts -> getAgentStepApprovalFingerprint(step, context, confirmation)
    - Creates a step-scoped approval fingerprint.
    - Prevents a single approval from approving all risky actions in the plan.

  src/content/content-script.ts -> buildAgentApprovalRequest(...)
    - Builds permission payload for the panel.
    - Includes prepared inputs, pending action, step fingerprint, resume step index, and previous action results.

Panel permission UI:
  src/devtools/panel/panelController.ts -> renderAgentPermissionMessage(message)
    - Shows prepared inputs.
    - Shows next action.
    - Explains that TestPilot will continue automatically after approval.
    - Buttons:
      Continue Once
      Allow Similar This Session
      Cancel

Approval execution:
  src/devtools/panel/panelController.ts -> approveAgentPermission(requestKey, options)
    - Sends approval payload back to content script.
    - Includes stepFingerprint, resumeFromStepIndex, and previousActionResults.

Resume behavior:
  src/content/content-script.ts -> runAgentWorkflow()
    - If approval is valid, it resumes at resumeFromStepIndex.
    - It does not rerun the whole plan from the top.
    - Previous safe actions are preserved in final result.

Session preference behavior:
  src/devtools/panel/panelController.ts -> agentApprovalPreferences
    - In-memory only for this panel session.
    - Allows similar non-destructive form-submit actions this session.
    - Scope includes page origin, page path, task type, action category, and target label.
    - High-risk actions are not rememberable.


Agent Live Event Flow
---------------------
src/content/content-script.ts emits Agent events through emitAgentEvent().

Events include:
  started
  plan-ready
  approval-granted
  step-started
  field-updated
  step-completed
  permission-required
  blocked
  completed

Panel handler:
  src/devtools/panel/panelController.ts -> handleAgentEvent(payload, sessionId)

Panel rendering:
  src/devtools/panel/panelController.ts -> renderAgentLiveMessage(message)
  src/devtools/panel/panelController.ts -> renderAgentPermissionMessage(message)
  devtools/panel.css -> .ai-chat-message.agent-live
  devtools/panel.css -> .ai-chat-message.agent-permission

Important UI behavior:
  Chat only sticks to bottom when the user is already near the bottom.
  If the tester scrolls up to read older output, new Agent updates should not pull them back down.


AI Chat Flow
------------
Entry point:
  src/devtools/panel/panelController.ts -> submitAiChat(value)

If current mode is Agent:
  submitAiChat() calls runAgentCommand(question)

If current mode is Chat:
  submitAiChat() calls completeChatWithSelectedProvider(question, options)

Local backend chat:
  src/devtools/panel/panelController.ts -> streamLocalBackendChat(question, options)
    POST /api/ai/chat-stream

Non-streaming backend fallback:
  src/devtools/panel/panelController.ts -> completeChatWithSelectedProvider()
    POST /api/ai/chat

Frontend provider prompt builder:
  src/devtools/panel/panelController.ts -> buildProviderChatMessages(question)
    System prompt:
      - You are TestPilot, a concise QA assistant inside Chrome DevTools.
      - Answer only from sanitized QA session context.
      - Do not invent bugs.
      - Never request or reveal API keys, cookies, passwords, or tokens.
    User payload:
      - question
      - compact sanitized context
      - responseStyle

Context builder:
  src/devtools/panel/panelController.ts -> buildCompactAiSessionSummary(task)
    - Uses buildAiSessionSummary(task).
    - Applies context mode limits: fast, balanced, deep.
    - Redacts sensitive data.
    - Includes latestAgentResult when available.


AI Artifact Flow
----------------
Buttons/menu actions:
  Analyze Session
  Generate Test Cases
  Generate Bug Report

Entry point:
  src/devtools/panel/panelController.ts -> runAiTask(task, options)

Task endpoints for local backend:
  analyze-session      -> POST /api/ai/analyze-session
  generate-test-cases  -> POST /api/ai/generate-test-cases
  generate-bug-report  -> POST /api/ai/generate-bug-report

Payload:
  { session: buildCompactAiSessionSummary(task) }

Frontend fallback/provider path:
  src/devtools/panel/panelController.ts -> completeAiTaskWithSelectedProvider(task)
  src/devtools/panel/panelController.ts -> buildProviderTaskMessages(task)

Task system prompt in frontend:
  src/devtools/panel/panelController.ts -> buildProviderTaskMessages(task)
    - Return only valid JSON.
    - Use sanitized session evidence only.
    - Do not invent product features or credentials.
    - Mark weak evidence as needs review.

Expected schema:
  src/devtools/panel/panelController.ts -> getAiAnalysisResponseSchema(task)

Validation:
  src/devtools/panel/panelController.ts -> validateAiAnalysis(value)
    - Normalizes model output.
    - Creates fallback test cases or bug drafts when needed.
    - Prevents empty/generic outputs from breaking UI.


Local AI Backend Flow
---------------------
Main file:
  ai-backend/src/server.ts

Runtime:
  Node.js 18+

Configuration:
  PORT
  AI_PROVIDER
  OLLAMA_BASE_URL
  OLLAMA_MODEL
  AI_MODEL_TIMEOUT_MS
  AI_CHAT_TIMEOUT_MS

Important note:
  createProvider() currently returns OllamaProvider only.
  Remote provider names are disabled in the backend and logged as disabled.

Health:
  GET /
    - Returns backend status JSON.

  GET /api/health
    - Calls Ollama /api/tags.
    - Returns provider/model health.

Chat:
  POST /api/ai/chat
  POST /api/chat
    - handleChat()
    - Uses buildChatSystemPrompt()
    - Uses provider.completeChat()
    - Validates answer relevance.
    - Falls back if model is unavailable or answer is not aligned.

Streaming chat:
  POST /api/ai/chat-stream
  POST /api/chat-stream
    - handleChatStream()
    - Uses provider.streamChat()
    - Sends server-sent events back to panel.

Analysis/test/bug generation:
  POST /api/ai/analyze-session
  POST /api/ai/generate-test-cases
  POST /api/ai/generate-bug-report
    - handleAi(req, res, task)
    - Sanitizes incoming session.
    - Builds deterministic fallback.
    - Calls provider.completeJson().
    - Normalizes/merges model result with fallback.

Backend prompt builder:
  ai-backend/src/server.ts -> buildSystemPrompt(task)
    - Tells model to use only sanitized summary.
    - Tells model not to invent issues.
    - Separates real issues from false positives.
    - Requires strict JSON shape.

Backend sanitization:
  ai-backend/src/server.ts -> sanitizeIncomingSession(session)
  ai-backend/src/server.ts -> redactText(text)
  ai-backend/src/server.ts -> redactSensitiveValue(value)

Provider implementation:
  ai-backend/src/server.ts -> OllamaProvider
    completeJson()
      - Calls Ollama OpenAI-compatible /v1/chat/completions.
      - Requests response_format json_object.

    completeChat()
      - Calls Ollama OpenAI-compatible /v1/chat/completions.

    streamChat()
      - Calls Ollama /api/chat with stream true.

Current Safety Boundaries
-------------------------
1. LLM does not execute browser actions.
2. Agent action execution is deterministic and validated in src/content/content-script.ts.
3. Approval is required before form submit/continue/save/send/login/signup/payment/destructive actions.
4. Approval is scoped to a request key and step fingerprint.
5. Approval resumes from the pending step, not from the beginning.
6. Session preference is in-memory only and scoped to page/task/action.
7. High-risk actions are not rememberable.
8. Sensitive values are redacted from AI context and persisted session data.
9. AI outputs are validated and fallback-generated when incomplete.


Known Extension Points for Senior Updates
-----------------------------------------
If adding LLM planning:
  - Add model planning in src/devtools/panel/panelController.ts or ai-backend/src/server.ts.
  - Keep src/content/content-script.ts as final validator.
  - Never execute model steps directly.
  - Normalize model steps through normalizeStructuredAgentPlan().
  - Keep ALLOWED_AGENT_ACTIONS limited.
  - Keep getAgentStepConfirmation() as the final permission gate.

If improving RAG/context:
  - Start with buildAiSessionSummary() and buildCompactAiSessionSummary().
  - Add retrieval/scoring before prompt construction.
  - Keep raw DOM, raw HTML, raw headers, cookies, and bodies out of context.
  - Prefer compact evidence chunks:
      finding title
      severity
      category
      route
      observed action
      linked API/console event
      recommendation

If improving Agent memory:
  - Use latestAgentResult and recentEvidenceEvents.
  - Keep memory scoped to current session/page.
  - Do not persist approval preferences across browser sessions without explicit settings UI.

If improving permissions:
  - Update getAgentStepConfirmation().
  - Update isAgentFormSubmitAction().
  - Update getAgentStepApprovalFingerprint().
  - Update renderAgentPermissionMessage().
  - Update tests/agent-live-workflow-smoke.js.

If improving prompts:
  - Frontend chat prompt: buildProviderChatMessages().
  - Frontend JSON task prompt: buildProviderTaskMessages().
  - Backend task prompt: ai-backend/src/server.ts buildSystemPrompt().
  - Backend chat prompt: ai-backend/src/server.ts buildChatSystemPrompt().
  - Schema: getAiAnalysisResponseSchema() in src/devtools/panel/panelController.ts and backend strict JSON shape in ai-backend/src/server.ts.

If improving generated artifacts:
  - Test cases: runAiTask('generate-test-cases'), validateAiAnalysis(), buildLocalTestCases().
  - Bug reports: runAiTask('generate-bug-report'), validateAiAnalysis(), buildLocalBugDrafts().
  - Report output: buildReport(), buildReportHtml(), buildReportMarkdown().

Manual browser validation:
  1. Reload extension in chrome://extensions.
  2. Open a staging form page.
  3. Open DevTools -> TestPilot.
  4. Start session.
  5. Switch to Agent.
  6. Ask Agent to test the form with valid dummy data.
  7. Confirm safe fields are filled visibly.
  8. Confirm Submit/Continue asks permission.
  9. Click Continue Once.
  10. Confirm Agent submits/continues automatically from that step.
  11. Confirm final response includes inputs used, action results, evidence, and next steps.
  12. Try a second risky action and confirm it asks again unless it is covered by a safe session preference.
